== Overview

This demo showcases Retrieval Augmented Generation (RAG), an AI framework that combines semantic search with language model generation to provide accurate, context-aware answers to questions.

This demo will:

* Install the required Stackable operators.
* Spin up the following data products:
** *OpenSearch*: A distributed search engine with k-NN vector search capabilities. This demo uses it to store document embeddings and perform semantic similarity searches.
** *OpenSearch Dashboards*: A visualization and user interface for OpenSearch. This demo uses it to inspect indexed documents and test queries.
** *Ollama*: A local LLM runtime. This demo uses it to run Mistral 7B (text generation) and nomic-embed-text (embeddings).
** *JupyterHub*: A multi-user Jupyter notebook server. This demo uses it to provide an interactive environment for exploring RAG functionality.
* Ingest sample documents about OpenSearch, RAG, AI, and the Stackable Platform with vector embeddings.
* Provide a pre-configured Jupyter notebook for interactive RAG queries.

The RAG workflow operates as follows:

. User submits a question
. Question is converted to a vector embedding
. OpenSearch performs k-NN search to find semantically similar documents
. Retrieved documents provide context for the language model
. Mistral 7B generates an answer based on the context and question

== What is RAG?

Retrieval Augmented Generation (RAG) addresses a key limitation of large language models: they can only generate responses based on their training data. RAG enhances LLMs by retrieving relevant information from a knowledge base in real-time and using it as context for generation.

Key benefits:

* *Up-to-date information*: Access current data not in the model's training set
* *Reduced hallucinations*: Answers grounded in retrieved facts
* *Transparency*: See which documents informed the answer
* *Domain-specific knowledge*: Use your own document corpus

== Understanding k-NN Vector Search

This demo uses {opensearch-knn}[OpenSearch's k-NN plugin] for semantic search.

=== How It Works

. Documents are converted to 768-dimensional vectors (embeddings) that capture semantic meaning
. Vectors are stored in OpenSearch with a special `knn_vector` field type
. At query time, your question is also converted to a vector
. OpenSearch uses the HNSW (Hierarchical Navigable Small World) algorithm to find the k nearest neighbor vectors
. Documents with similar vectors are semantically similar, even without keyword matches

=== Index Configuration

The demo creates an index with this k-NN mapping:

[source,json]
----
{
  "mappings": {
    "properties": {
      "embedding": {
        "type": "knn_vector",
        "dimension": 768,
        "method": {
          "name": "hnsw",
          "space_type": "cosinesimil",
          "engine": "nmslib"
        }
      }
    }
  }
}
----

* *dimension*: Vector size (768 for nomic-embed-text)
* *space_type*: Distance metric (cosine similarity for text)
* *hnsw*: Approximate nearest neighbor algorithm (fast and accurate)

== Architecture Details

=== Components

*OpenSearch*::
* Single-node cluster with k-NN plugin enabled
* Stores documents with vector embeddings
* Performs semantic similarity search
* Resources: 2 CPU, 4 GB RAM

*Ollama*::
* Runs two models:
** *nomic-embed-text*: Generates 768-dim embeddings (~274MB)
** *mistral:7b*: 7 billion parameter LLM for generation (~4.1GB)
* Resources: 2-4 CPU, 4-8 GB RAM
* Persistent storage: 15 GB for models

*JupyterHub*::
* Multi-user notebook environment
* Spawns individual pods per user
* Pre-configured with OpenSearch and Ollama clients
* Resources per user: 1-2 CPU, 2-4 GB RAM

*OpenSearch Dashboards*::
* Web UI for OpenSearch
* Index management and query testing
* Document inspection

=== Data Flow

[source,text]
----
User Query
    ↓
nomic-embed-text (Ollama)
    ↓
Query Vector [768 dims]
    ↓
OpenSearch k-NN Search
    ↓
Top-k Similar Documents
    ↓
Mistral 7B (Ollama) + Context
    ↓
Generated Answer
----

== Customization

=== Using Different Models

Edit `stacks/opensearch-rag/ollama.yaml` to use alternative models:

[source,yaml]
----
ollama:
  models:
    - nomic-embed-text  # Or: all-MiniLM-L6-v2
    - llama3.2          # Or: phi3, gemma
----

NOTE: If you change the embedding model, ensure the dimension in the index mapping matches the model's output size.

=== Adding Your Own Documents

From the Jupyter notebook, you can add custom documents:

[source,python]
----
# Your document
doc = {
    'title': 'My Document',
    'content': 'Document content...',
    'category': 'custom',
    'embedding': get_embedding('Document content...'),
    'timestamp': '2025-01-01T00:00:00'
}

# Index it
opensearch_client.index(
    index='rag-documents',
    body=doc
)
----

=== Tuning Retrieval

Adjust the number of retrieved documents:

[source,python]
----
# More context = better answers but slower
result = rag_query("Your question", k=5)

# Less context = faster but may miss relevant info
result = rag_query("Your question", k=1)
----

== Troubleshooting

=== Ollama Models Not Loading

If you see "model not found" errors:

[source,console]
----
# Check if models are pulled
$ kubectl exec deployment/ollama -- ollama list

# Manually pull if needed
$ kubectl exec deployment/ollama -- ollama pull nomic-embed-text
$ kubectl exec deployment/ollama -- ollama pull mistral:7b
----

=== Data Ingestion Failed

[source,console]
----
# Check job logs
$ kubectl logs job/opensearch-rag-ingestion

# Delete and retry
$ kubectl delete job opensearch-rag-ingestion
$ kubectl apply -f https://raw.githubusercontent.com/stackabletech/demos/main/demos/opensearch-rag/data-ingestion.yaml
----

=== Slow Response Times

The first query to Mistral 7B may be slow as the model loads into memory. Subsequent queries should be faster.

Consider:

* Reducing context window (use fewer documents)
* Using a smaller model (llama3.2-1b)
* Increasing Ollama pod resources

=== OpenSearch Connection Issues

[source,console]
----
# Verify OpenSearch is running
$ kubectl get pods -l app.kubernetes.io/name=opensearch

# Test connectivity from Jupyter pod
$ kubectl exec jupyter-<username> -- curl -k https://opensearch:9200 -u admin:adminadmin
----
